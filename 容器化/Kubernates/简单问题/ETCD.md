# 简述ETCD及其特点？

**思路**：

- 定位：etcd 是 Kubernetes 的“唯一真相源”；
- 技术本质：分布式、强一致、高可用的键值存储；
- 核心特性：基于 Raft 共识算法、支持 Watch 机制、轻量高效；
- 与 K8s 的关系：存储所有集群状态（如 Pod、Service、Node 等元数据）。

**回答**：

etcd 是一个开源的分布式键值存储系统，专为高可用和强一致性场景设计，Kubernetes 使用它作为**唯一的集群状态存储后端**。其核心特点包括：

- 基于 **Raft 共识算法**，确保多节点间数据强一致；
- 支持 **Watch 机制**，客户端可监听 key 变化，实现事件驱动（如控制器响应资源变更）；
- 提供简单 HTTP/JSON API，读写性能高、延迟低；
- 数据持久化、支持 TLS 加密与 RBAC 访问控制；
- 轻量级、无依赖，适合存储小规模但关键的元数据（如配置、状态、锁信息）。



# 简述ETCD适应的场景？

**思路**：

从“强一致 + 高可用 + 小数据量”出发，列举典型用途，突出其在协调服务中的角色，而非通用数据库。

**回答**：

etcd 适用于对**数据一致性、可靠性和低延迟读写**要求高，但数据规模不大的场景，典型包括：

- **Kubernetes 集群元数据存储**（如 Pod、Deployment、Service 等对象状态）；
- **分布式系统协调**（如选主、分布式锁、服务发现）；
- **配置管理**（集中式动态配置中心，支持实时推送变更）；
- **任务调度系统**的状态同步（如 Apache Kafka Controller 选举）。

> 注意：etcd 不适合存储大文件、日志或高频写入的大数据量场景，应作为“控制平面”的状态引擎，而非数据平面存储。
