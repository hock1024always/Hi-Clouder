# 简述什么是Kubernetes？

回答要点：

1. **架构核心**——主从式（控制平面 + 工作节点）；
2. **核心用途**——自动化部署、扩缩容、自愈、服务发现等容器编排能力；
3. **常见组件**——列举关键组件如 Pod、Deployment、Service、kube-scheduler、kubelet 等，体现对系统结构的理解。

Kubernetes 是一个主从架构的容器编排系统，由控制平面（含 API Server、etcd、kube-scheduler、kube-controller-manager）和工作节点（运行 kubelet、kube-proxy 及容器运行时）组成，用于自动化管理容器化应用的部署、扩缩容、自愈和服务发现；其核心对象包括 Pod（最小调度单元）、Deployment（管理副本与滚动更新）、Service（提供稳定网络访问）等，通过声明式 API 实现应用的可靠运行与运维自动化。

# 简述Kubernetes的优势、适应场景及其特点？



- Kubernetes 的优势在于提供强大的自动化运维能力，包括自动扩缩容、故障自愈、滚动更新和资源调度，显著提升系统可靠性和运维效率；
- 它特别适用于微服务架构、持续交付（CI/CD）、多环境（开发/测试/生产）统一管理以及混合云或多云部署等场景；
- 其核心特点包括声明式配置（通过 YAML 定义期望状态）、松耦合的模块化架构、丰富的扩展机制（如 CRD、Operator）、跨基础设施的可移植性，以及围绕 Pod、Service、Namespace 等抽象构建的标准化应用模型，使应用在不同环境中具备一致的运行与管理体验。



# 简述Kubernetes的缺点或当前的不足之处？

- 学习曲线陡峭，对初学者和小团队门槛较高；
- 部署和运维复杂，尤其在自建集群时需维护高可用控制平面；
- 资源开销较大（如 etcd、sidecar 容器等），对小型应用可能“杀鸡用牛刀”；
- 调试和排障困难，日志、网络、权限等问题链路长；
- 此外，其声明式模型虽强大，但若缺乏规范易导致配置冗余或状态漂移。对于简单单体应用或低频服务，引入 K8s 可能带来不必要的复杂性。

# 简述Kubernetes相关基础概念？

选取最核心、高频使用的抽象对象，按逻辑层次组织：从最小单元到工作负载再到服务暴露与配置管理。

- **Pod**（最小调度和运行单元，包含一个或多个共享网络/存储的容器）；
- **Deployment**（声明式管理 Pod 副本数、滚动更新与回滚）；
- **Service**（为 Pod 提供稳定的网络访问入口，支持 ClusterIP、NodePort、LoadBalancer 类型）；
- **Namespace**（逻辑隔离多租户或多环境资源）；
- **ConfigMap 和 Secret**（分别用于非敏感和敏感配置数据的解耦管理）；
- 以及 **Label/Selector**（通过键值标签实现资源分组与选择）。这些构成了应用部署与管理的基本语义。



# 简述Kubernetes集群相关组件？

按控制平面（Control Plane）和工作节点（Worker Node）两大角色分类，列出关键组件及其职责，突出协同关系。

Kubernetes 集群由控制平面和工作节点组成。

- **控制平面**包括：
  - **kube-apiserver**（集群统一入口，处理 REST 请求）、
  - **etcd**（分布式键值存储，保存集群状态）、
  - **kube-scheduler**（负责 Pod 调度到合适节点）、
  - **kube-controller-manager**（运行各类控制器如 Deployment、Node 控制器）；
- **工作节点**包括：
  - **kubelet**（管理本机 Pod 和容器生命周期）、
  - **kube-proxy**（实现 Service 网络代理和负载均衡）、
  - **容器运行时**（如 containerd，负责运行容器）。

这些组件协同工作，实现集群的自动化管理。

# 简述Kubernetes和Docker的关系？

Docker 是容器运行时之一，K8s 是编排平台；强调 K8s 已解耦对 Docker 的依赖。

Kubernetes 是容器编排平台，而 Docker 是一种容器运行时技术。

- 早期 Kubernetes 默认使用 Docker 作为底层容器运行时，
- 但自 v1.20 起已弃用 Docker Engine（dockershim），转而原生支持符合 CRI（容器运行时接口）标准的运行时（如 containerd、CRI-O）。

- 如今 Docker 主要用于本地开发构建镜像，而生产环境中 K8s 通常直接对接 containerd 等轻量级运行时。因此，二者是“编排 vs 运行”的上下游关系，而非绑定依赖。
