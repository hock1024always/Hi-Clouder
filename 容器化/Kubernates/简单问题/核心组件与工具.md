# 简述Kubernetes中什么是Minikube、Kubectl、Kubelet？

定位+用途：Minikube 是本地开发工具，kubectl 是命令行客户端，kubelet 是节点代理。避免混淆角色（如别把 kubectl 当服务端组件）

- **Minikube**：一个用于在本地单机（如笔记本）快速部署单节点 Kubernetes 集群的工具，适合开发和测试；
- **Kubectl**：Kubernetes 的官方命令行客户端，通过与 API Server 交互，用于管理集群资源（如创建 Pod、查看日志等）；
- **Kubelet**：运行在每个工作节点上的核心代理，负责监听分配给该节点的 Pod，并确保容器按期望状态运行（如启动、健康检查、上报状态）。



# 简述Kubernetes各模块如何与API Server通信？

**思路**：

强调 API Server 是唯一入口，其他组件通过 HTTPS + 认证（如 TLS、ServiceAccount）与其通信；区分“主动调用”和“监听变化”两种模式（即 REST 请求 vs Watch 机制）。

**回答**：

Kubernetes 所有组件（如 kube-scheduler、kube-controller-manager、kubelet、kube-proxy）均通过 **HTTPS 协议**与 API Server 通信。它们使用 TLS 证书或 ServiceAccount Token 进行身份认证和授权。通信方式主要有两种：

1. **主动请求**：如 kubelet 定期上报节点状态（POST/PUT）；
2. **Watch 机制**：如 kube-scheduler 监听未绑定的 Pod（通过长连接 watch），一旦有新 Pod 就触发调度。
   API Server 是集群状态的唯一权威源，所有读写操作必须经由它完成。



# 简述Kubernetes kubelet的作用？

**思路**：

聚焦 kubelet 在节点上的核心职责：Pod 生命周期管理、健康探针执行、资源上报、卷挂载等，体现其“节点管家”角色。

**回答**：

Kubelet 是运行在每个工作节点上的核心代理，主要作用包括：

- 从 API Server 获取分配给本节点的 PodSpec；
- 调用容器运行时（如 containerd）创建/销毁容器；
- 执行 Liveness/Readiness 探针，监控容器健康状态；
- 挂载 Volume（如 ConfigMap、PVC）到 Pod；
- 定期向 API Server 上报节点和 Pod 状态（如 CPU、内存使用情况）。
  它是确保节点上容器按声明状态运行的关键组件。
